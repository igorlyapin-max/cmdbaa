# E2E scenario: template, verification contracts, external verification

## Статус

Сценарий описывает минимальную ручную проверку полного контура BAA. Реальный
endpoint `cmdbcustompage` пока не готов, поэтому автоматический smoke-level тест
для внешней верификации откладывается. До появления endpoint проверяются шаги
подготовки, публикации contracts, сохранения endpoint definition и UI-логика
интерпретации.

## Предусловия

- Reverse proxy BAA запущен на `127.0.0.1:8094`.
- Пользователь вошел в CMDBuild и имеет действующую сессию.
- Схема BAA создана в `Настройки / Схема`.
- В схеме присутствуют классы:
  - `BAAConversionContract`;
  - `BAAConversionContractVersion`;
  - `BAAVerificationInputContract`;
  - `BAAVerificationOutputContract`;
  - `BAAVerificationEndpoint`.
- В `BAAVerificationEndpoint` есть атрибут `ResultInterpretationJson`.
- Есть VSDX-шаблон и соответствующий файл контрольной суммы.

## Ручной сценарий

1. Открыть BAA custom page.
2. Нажать `Загрузить шаблон/схему` и выбрать VSDX.
3. Если используется checksum, загрузить файл контрольной суммы.
4. Открыть `Контракты`.
5. Создать или выбрать conversion contract.
6. Открыть `Подготовить шаблон`.
7. В блоке `Обогатить` привязать объект контракта на схеме.
8. Назначить CMDB-классы и атрибуты на нужные визуальные типы.
9. При необходимости настроить `Отразить на связь`.
10. Нажать `Сохранить контракт`.
11. Нажать `Сохранить шаблон`.
12. Открыть `Проверить шаблон`.
13. Нажать `Проверить шаблон`.
14. Убедиться, что техническая проверка показывает успешный статус или понятные
    ошибки для исправления.
15. Открыть `Подготовить объекты`.
16. Нажать `Сформировать план`.
17. Если есть незаполненные обязательные значения, выполнить
    `Дозаполнить через контракт` или `Дозаполнить руками`.
18. Сохранить правила дозаполнения через `Сохранить контракт`, если они должны
    стать частью версии контракта.
19. Открыть `Подготовить правила верификации`.
20. В блоке `Контракты верификации` нажать `Сформировать по готовым объектам`.
21. Проверить, что сформированы input/output contracts.
22. Нажать `Опубликовать готовые объекты в CMDBuild`.
23. Проверить, что опубликованные input/output contracts имеют статус `Active`.
24. Оставаться в `Подготовить правила верификации`.
25. Выбрать `Input contract` и `Output contract`.
26. Заполнить `Код endpoint`.
27. Заполнить `URL endpoint` абсолютным URL с протоколом и host. Пока реальный
    endpoint не готов, URL может быть сохранен как проектный placeholder в
    формате абсолютного URL, например
    `https://cmdb.example.org/cmdbuild/custompage/api/verify/network-acl`.
28. Заполнить `Params JSON`, если endpoint требует параметры вызова.
29. Настроить `Интерпретация результата`:
    - выбрать режим, например `строки есть = ошибка`;
    - выбрать `Все таблицы` или конкретный `Table code`;
    - заполнить сообщения для найденных и ненайденных данных;
    - выбрать, когда показывать таблицы результата.
30. Установить статус endpoint `Active`.
31. Нажать `Сохранить endpoint`.
32. Нажать `Обновить endpoint` и убедиться, что endpoint definition появился в
    списке.
33. Выбрать сохраненный endpoint и проверить, что URL, contracts, params и
    интерпретация подставились обратно в форму.
34. Открыть `Верификация`.
35. Проверить, что для запуска доступен выбор сохраненного endpoint без
    редактирования URL, contracts, params и интерпретации.

## Проверка внешней верификации после готовности endpoint

Когда тестовый endpoint будет готов:

1. Открыть `Верификация`.
2. Выбрать сохраненный `Active` endpoint.
3. Нажать `Выполнить верификацию`.
4. Проверить, что BAA:
   - сверил текущий план с input contract;
   - вызвал endpoint методом `POST`;
   - провалидировал ответ по output contract;
   - применил `ResultInterpretationJson`;
   - показал итог интерпретации и таблицы результата.
5. Для режима `rows_present_is_error` проверить два ответа endpoint:
   - пустая таблица: интерпретация `passed`, создание не блокируется;
   - таблица с одной строкой: интерпретация `failed`, создание блокируется.
6. Для режима `manual_review` проверить, что таблицы показываются, но создание
   не блокируется жестко.
7. Для технической ошибки endpoint проверить, что интерпретация
   `technical_error`, а создание заблокировано.

## Ожидаемые результаты

- Контракт и версия контракта сохраняются в CMDBuild.
- Обогащенный VSDX содержит `_baa_` metadata и пользовательские `template_`
  поля.
- План объектов можно открыть повторно без потери контекста.
- Input/output verification contracts публикуются в CMDBuild.
- Endpoint definition сохраняется вместе с `ResultInterpretationJson`.
- Результат внешней верификации влияет на `Создать объекты` только через
  интерпретацию BAA:
  - `failed` и `technical_error` блокируют создание;
  - `passed`, `warning`, `manual_review` не блокируют создание.

## Будущий smoke-level автотест

После появления тестового endpoint рекомендуется добавить автоматическую
проверку, которая возвращает:

- успешный ответ с пустой таблицей;
- успешный ответ с одной строкой;
- техническую ошибку `success: false`;
- legacy-ответ только с `items`.

Автотест должен проверять итоговую интерпретацию BAA и состояние create gate.
