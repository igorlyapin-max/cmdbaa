# CMDB BAA: помощь пользователя

## Назначение

CMDB BAA подготавливает Visio VSDX-шаблон для последующего создания объектов в
CMDBuild. Пользовательский сценарий состоит из семи рабочих блоков:

- `Контракты` - создание и просмотр контрактов конвертации.
- `Подготовить шаблон` - загрузка VSDX, назначение классов и атрибутов CMDB,
  обогащение файла Shape Data.
- `Проверить шаблон` - техническая проверка BAA-полей и версии контракта.
- `Подготовить объекты` - построение dry-run плана и дозаполнение.
- `Подготовить правила верификации` - настройка endpoint, contracts,
  параметров вызова и интерпретации результата.
- `Верификация` - бизнес-проверки заполненного шаблона.
- `Создать объекты` - запись подготовленного плана в CMDBuild.

В верхней панели расположены общие действия:

- `Загрузить шаблон/схему` - загружает рабочий VSDX для всех блоков меню.
- `Сохранить контракт` - сохраняет изменения контракта. В блоке `Подготовить
  объекты` сохраняет правила дозаполнения класса и перестраивает план.
- `Сохранить шаблон` - сохраняет обогащенный VSDX и файл контрольной суммы.

## Контракты

Контракт конвертации описывает, как BAA распознает Visio-типы и какие CMDB
классы/атрибуты должны быть записаны в шаблон. Пользователь выбирает контракт,
а версии создаются автоматически.

Версия контракта создается, когда при обогащении меняется состав известных
типов, агрегатов, маппингов, параметров контракта или правил отображения
атрибутов. Если пользователь расширяет ранее версионированный шаблон, BAA
создает новую версию. Удаление типов из конкретного VSDX не инвалидирует
контракт.

Контракт хранится в CMDBuild в технических классах BAA. В VSDX фиксируется
ссылка на версию контракта и checksum правил. Если объект контракта или версия
контракта удалены, проверка и конвертация невозможны.

## Подготовка шаблона

На странице `Подготовить шаблон` пользователь загружает VSDX и назначает:

- объект контракта в самой схеме;
- CMDB-классы на Visio-типы;
- атрибуты классов, которые должны появиться в Shape Data;
- параметры контракта;
- правила отображения атрибутов связи на атрибуты endpoint-объектов.

BAA записывает только свои поля:

- технические поля с префиксом `_baa_`;
- пользовательские поля шаблона с префиксом `template_`.

Чужие Shape Data не переименовываются и не перезаписываются.

## Объект маппинга

`source`, `destination` и `relation` в выражениях относятся не к произвольному
объекту на схеме, а к объекту маппинга.

Для обычной фигуры объект маппинга - это сама фигура. Для соединительной линии
объект маппинга состоит из трех частей:

- `source` - endpoint на исходной стороне линии;
- `destination` - endpoint на конечной стороне линии;
- `relation` - сама соединительная линия.

Именно объект маппинга определяет, откуда брать значения атрибутов при создании
экземпляра CMDB-класса.

Поддерживаемые выражения:

```text
${visioparam.name}
${source.visioparam.name}
${destination.visioparam.name}
${relation.visioparam.name}
${contractparam.name}
```

На странице `Подготовить объекты` поля дозаполнения поддерживают подсказки. Начните
вводить `$`, `${`, `${visi`, `${source` или `${contract`, выберите вариант
стрелками и нажмите `Enter`/`Tab`.

Для `Code` и `Description` подсказки работают как конструктор составной строки
по текущему объекту маппинга. Это позволяет строить значения из полей source,
destination и relation текущего экземпляра, например:

```text
${source.visioparam.ipaddress}-${destination.visioparam.ipaddress}
${source.visioparam.ipaddress}-${destination.visioparam.ipaddress}-${relation.visioparam.Protocol}
```

В списке подсказок длинные роли показываются компактно:

```text
src ipaddress  -> ${source.visioparam.ipaddress}
dst ipaddress  -> ${destination.visioparam.ipaddress}
rel Protocol   -> ${relation.visioparam.Protocol}
self Name      -> ${visioparam.Name}
```

Для ручного дозаполнения подсказки ограничены конкретным объектом плана. Для
дозаполнения через контракт подсказки ограничены текущим классом и объектом
маппинга этого класса.

## Агрегации Visio

BAA различает группы и контейнеры. По умолчанию группа или контейнер
рассматриваются как единый визуальный объект: атрибуты записываются на один
якорный элемент, а не декомпозируются во все внутренние фигуры.

Декомпозиция групп и контейнеров включается в настройках типов. Если
декомпозиция включена, пользователь может назначать классы на внутренние
атомарные фигуры.

Группы не инвалидируют связи. Многие Visio-фигуры, выглядящие атомарно для
пользователя, технически являются группами. Поэтому BAA не запрещает такие
связи, но выводит предупреждения, если endpoint нельзя однозначно сопоставить
с CMDB-назначением.

## Висящие связи

Соединительная линия считается висящей, если не определен source или
destination endpoint. Такие связи не удаляются из плана молча:

- `unbound` - не привязаны оба конца;
- `partial` - привязан только один конец;
- предупреждение - endpoint является группой или требует fallback-разрешения.

Техническая проверка и план создания показывают каждую такую связь. Реальное
создание блокируется, если из-за связи невозможно собрать обязательные
атрибуты.

## Обогащение

Обогащение создает новую или переиспользует существующую версию контракта,
записывает в VSDX `_baa_` metadata и добавляет `template_` поля для
пользовательского заполнения.

Lookup/reference атрибуты могут наполняться списками:

- постоянный список - значения записываются в шаблон;
- переменный список - значение выбирается или вводится позже.

Для reference действует лимит количества объектов. Если объектов больше
настройки `Максимум объектов reference для постоянного списка`, BAA переводит
список в переменный режим и сообщает пользователю.

## Подготовка и создание объектов

`Подготовить объекты` сначала строит план. План можно открыть повторно кнопкой
`План объектов` без пересборки.

Дозаполнение бывает двух типов:

- `Дозаполнить через контракт` - правило применяется ко всем экземплярам класса
  и сохраняется в новой версии контракта.
- `Дозаполнить руками` - значение применяется только к конкретному объекту
  текущей сессии.

После `Сохранить контракт` BAA сразу использует новую версию контракта в текущей
сессии. Чтобы эта версия была зафиксирована внутри VSDX и открывалась так же в
следующий раз, нужно выполнить `Сохранить шаблон`.

Если по одному `класс + атрибут` одновременно есть ручное значение и правило
контракта, строка помечается желтым предупреждением.

Красным выделяются только ошибочные сущности: обязательные атрибуты, для
которых нет значения или правила заполнения.

После подготовки и верификации отдельный пункт меню `Создать объекты` отправляет
в CMDBuild payload из последнего подготовленного плана. Исполнитель выбирает
полноту запуска в одном из двух режимов:

- `Выбрать классы` - создать все объекты выбранных CMDB-классов.
- `Выбрать объекты` - создать только отмеченные строки плана.

Обязательность и блокирующие ошибки проверяются для выбранной части плана. Если
файл контрольной суммы не загружен или не совпадает с VSDX, создание объектов не
выполняется.

## Внешняя верификация

BAA может передать подготовленный план во внешний endpoint `cmdbcustompages`.
Перед использованием проверки должны быть опубликованы contracts:

- input contract - какие классы, атрибуты и связи BAA отправляет;
- output contract - какой формат ответа BAA ожидает.

Contracts формируются в меню `Подготовить правила верификации`, блок
`Контракты верификации`, и публикуются в CMDBuild. Кнопки явно работают с
готовыми объектами: в input contract и во внешний вызов попадают только объекты
плана без незаполненных обязательных значений и без блокирующих ошибок. В этом
же блоке выводятся списки опубликованных input и output contracts с кодом,
версией, статусом и checksum. После этого
администратор `cmdbcustompages` настраивает endpoint проверки. В меню
`Подготовить правила верификации` пользователь выбирает опубликованные Active
input/output contracts, выбирает или сохраняет endpoint definition, указывает
endpoint URL, параметры вызова и правило интерпретации результата. Endpoint
можно сохранить со статусом `Active`, `Draft` или `Archived`, но запуск
выполняется только по `Active`.

BAA не строит бизнес-логику проверки из данных динамически. Он только отправляет
подготовленный JSON и отображает короткий итог ответа endpoint. Внешний
`cmdbcustompage` возвращает найденные данные в виде таблиц. BAA по настройке
`Интерпретация результата` решает, что означает наличие или отсутствие строк:
ошибку, предупреждение, ручной анализ или только технический успех.

Перед вызовом endpoint BAA локально сверяет текущий план с выбранным input
contract. Если в плане нет требуемого класса, атрибута или relation context,
endpoint не вызывается. Ответ endpoint валидируется по выбранному output
contract. Если endpoint вернул таблицы, BAA показывает их согласно настройкам
`Показывать таблицы при срабатывании` и `Показывать таблицы если не сработало`.

В меню `Верификация` пользователь выбирает уже сохраненный `Active` endpoint и
запускает проверку. Ручное редактирование URL, contracts, params и
интерпретации вынесено из этого меню в `Подготовить правила верификации`.

Если интерпретация результата дала `failed` или `technical_error`, пункт
`Создать объекты` блокирует создание до успешного повторного запуска
верификации. Режимы `warning` и `manual_review` не блокируют создание, но
выводятся оператору. При перестроении плана результат предыдущей внешней
проверки сбрасывается, потому что изменился набор данных для проверки.

Пошаговый E2E-сценарий проверки полного контура описан в
`docs/e2e-verification-scenario.md`. Пока тестовый `cmdbcustompage` endpoint не
готов, внешний вызов проверяется только до сохранения endpoint definition и
настроек интерпретации.
